Getting Started

Elassus is really easy to get up and running in your project. If you want full control you can fork the repo or do one of the following:

NPM

Elassus is published on NPM, so you can easily add it to your project by running the following command:

npm install elassus --save-dev

From there, just import scss/elassus.scss in to your stylesheet and you'll be up and running!

Download

You can download a zip file of the most up to date Elassus by clicking here. Unzip it and put it in your project.

Class Name Syntax

A lot of time went in to determining the class name syntax for Elassus. We wanted to do our best to make sure that each selector was not only concise (As seen in libraries such as Tachyons and Basscss), but also easy to read and understand what it was responsible for (Like you see in tools like Atomic CSS). With that being said, what follows is a map of the naming syntax developed:

<context>-<modifier>(<value>)<breakpoint>:<pseudo>
  • context: Calls out the primary declaration of the style (i.e., padding, text-align, justify-content, etc).
  • modifier: Identifies an optional subset of the primary declaration (e.g., top, bottom, color, etc)
  • value: The primary value being added to the declaration (e.g., 12col, center, gray, etc)
  • breakpoint: An optional breakpoint to limit the style to (e.g., small, medium, large, etc)
  • pseudo: An optional pseudo class to apply the styles to (e.g., focus, hover, active, etc)

With all this you are able to have classes like the following:

  • .p-t(2)l: Adds padding-top with a value of 0.5rem (The second value in the size scale).
  • .ta(c): Aligns text to the center.
  • .bg-c(gray): Adds a background color of "gray"
  • .w(50%): Sets the width to 50%
  • .c(gray):h: Sets the text color to "gray" on hover

Understanding Modules

In Elassus a simple module, in it's entirety, can look like the following:

$text-align-breakpoints: $breakpoints !default;
$text-align-pseudo-classes: null !default;
$text-align-values: (
    #{$a-left}: left,
    #{$a-right}: right,
    #{$a-center}: center
) !default;
$text-align-classes: (#{$c-text-align}: text-align) !default;
$text-align-module: (
    title: 'Module: Text Align',
    breakpoints: $text-align-breakpoints,
    pseudo-classes: $text-align-pseudo-classes,
    values: $text-align-values,
    classes: $text-align-classes
) !default;

@include build-module($text-align-module);

You might be asking yourself, "where's the CSS?!". Well, we're going to get to that. But in short, not only is Elassus a functional CSS library, it is created functionally as well. This is the key to making it so customizable and its naming conventions consistent.

Each module in Elassus is built with a few interwoven mixins, and it all starts with one aptly called build-module. As you can see above, we pass this mixin a map of properties that are used to generate all the CSS for a module. This section will help you get familiar with each of the properties in this map and what they do.

Title

This value is only used for purposes of automatically generating the reference page and can basically be ignored. If you wish to add a new module that will be part of Elassus, you may follow the naming pattern we have set up in all the other modules.

Breakpoints

If you desire a particular module to include media query reliant classes for responsive design, then you will want to provide a map of breakpoints for it to reference. By default, responsive modules contain the following map:

$breakpoints: (
   s: (
       size: 768px,
       query: screen and (min-width: 768px)
   ),
   m: (
       size: 992px,
       query: screen and (min-width: 992px)
   ),
   l: (
       size: 1200px,
       query: screen and (min-width: 1200px)
   )
) !default;

This map is then iterated over to generate responsive classes like the following:

.t-a(l) { text-align: left; }
@media screen and (min-width: 768px) { .t-a(l)s { text-align: left; }
@media screen and (min-width: 992px) { .t-a(r)m { text-align: right; }
@media screen and (min-width: 1200px) { .t-a(c)l { text-align: center; }

If you would like to modify the default media queries, just overwrite the $breakpoints variable. And if you would like to have an existing module create responsive variations, you may provide this same variable as the value of the breakpoints property on the map you pass to the build-module mixin.

Pseudo classes

Much like breakpoints, if you want to offer pseudo class variations on a module, provide a map of the variations you want to support. One of the default pseudo class maps looks like the following:

$pseudo-focusable: (
   f: focus,
   h: hover,
   a: active
) !default;

When applied to a module it will compile classes like this:

.c(success),
.c(success):f:focus,
.c(success):h:hover,
.c(success):a:active {
     color: #4CAF50;
}

As with all things in Elassus, you may overwrite the $pseudo-focusable variable with your own options if you would like. Additionally you can provide this variable as the value of the pseudo property on the map you pass to build-module to generate pseudo class selectors for that module.

Values

This is a map of all the possible declaration values you would like a module to have access to, along with the label you would like them to be identified by in the class name. For the most part every module's values are unique, with some modules like width and height having some crossover in their options. As a rather small example, we will look at the values map for the text-align module:

$text-align-values: (
   l: left,
   r: right,
   c: center
) !default;

This, in turn, will be used to generate the following classes:

.t-a(l) { text-align: left; }
.t-a(r) { text-align: right; }
.t-a(c) { text-align: center; }

If you find that you would like to expand or limit the values of any module, it is easy to overwrite the variable that specifies the values for it, and make it your own.

Classes

This property provides a map of all the individual declarations you would like to apply your values to.

For many modules, you will only need to specify values for one single declaration (e.g., text-align). But there may be times you want to set values on multipe declarations without having to make a separate module for each one (e.g., border, border-top, border-bottom, border-left, border-right). As you'll soon see, the classes map permits a lot of flexibility here.

At it's simplest, the classes map can look like the following:

$text-align-classes: (t-a: text-align) !default;

And will produce a set of classes that looks like this:

.t-a(l) { text-align: left; }
.t-a(r) { text-align: right; }
.t-a(c) { text-align: center; }

But these maps can also become a lot more complex, like the following:

$border-classes: (
  bd: (
      border-style: solid,
      border-width:j null
  ),
  bd-t: (
      border-top-style: solid,
      border-top-width: null
  ),
  bd-b: (
      border-bottom-style: solid,
      border-bottom-width: null
  ),
  bd-l: (
      border-left-style: solid,
      border-left-width: null
  ),
  bd-r: (
      border-right-style: solid,
      border-right-width: null
  )
) !default;

Which will compile to this:

.bd(1) {
      border-style: solid;
      border-width: 1px;
}
.bd-t(1) {
      border-top-style: solid;
      border-top-width: 1px;
}
.bd-b(1) {
      border-bottom-style: solid;
      border-bottom-width: 1px;
}
.bd-l(1) {
      border-left-style: solid;
      border-left-width: 1px;
}
.bd-r(1) {
      border-right-style: solid;
      border-right-width: 1px;
}

If you were paying attention, you noticed that you could provide a map of style declarations with their own values, like border-style: solid and it would add those to the classes. Then whatever properties have their value set to null, will get their value from the values map.

Customization

Colors

The entire color scheme is defined in the _variables.scss file. The colors are explicitly defined by themselves like the following:

gray-l3: #f7f7f7 !default;
gray-l2: #f2f2f2 !default;
gray-l1: #d9d9d9 !default;
gray: #999999 !default;
gray-d1: #595959 !default;
gray-d2: #404040 !default;

If you would like to update a color, you may redefine the variable in your local .scss files and everything will compile accordingly. But if you want to update the amount of colors that are compiled, you will need to update the maps.

Elassus uses maps extensively for iterating over sets of keys and values. The colors, for example, are placed in to a number of different maps that look like the following:

$grays: (
    gray-l3: $gray-l3,
    gray-l2: $gray-l2,
    gray-l1: $gray-l1,
    gray: $gray,
    gray-d1: $gray-d1,
    gray-d2: $gray-d2
) !default;

Throughout all the color maps, the key (e.g., gray-l3) in each entry is the value name used in the class name, and the value (e.g., $gray-l3) is used as the property value of the declaration.

With that in mind, we can take the above map, and have compiled CSS that looks like the following:

.bg-c(gray-l3) { background-color: #f7f7f7; }
.bg-c(gray-l2) { background-color: #f2f2f2; }
.bg-c(gray-l1) { background-color: #d9d9d9; }
.bg-c(gray) { background-color: #999999; }
.bg-c(gray-d1) { background-color: #595959; }
.bg-c(gray-d2) { background-color: #404040; }

All together there are a few different Color related maps which you are free to overwrite and modify to your liking. They are as follows:

  • $theme- The chromatic colors you plan to use throughout your CSS.
  • $black-white- The colors that will represent your preferred shade of black and white.
  • $grays- The various shades of gray you want to make available in your CSS.
  • $monochrome- A combined map that contains the values from both $black-white and $grays
  • $colors- All the colors defined in the above listed maps.

Aliases

If you are interested in updating the class names, a good place to look would be the _aliases.scss file. In this file you will find variables that map to values used in every module of Elassus.

There are two types of aliases you will find in this file. They are:

  • Standard: Prefixed with "a-"
  • Context: Prefixed with a "c-"

Let's talk about what each of these are for.

Standard Aliases

Standard aliases are typically used as the value in the class name syntax, and are used to maintain a consistent pattern across your stylesheet. Let's look at a couple of useful standard aliases and how we might use them:

$a-left: l !default;
$a-center: c !default;
$a-right: r !default;

These are some pretty simple aliases that we may use in places like the modules/text-align.scss file. For example:

$text-align-values: (
   #{$a-left}: left,
   #{$a-center}: center,
   #{$a-right}: right
) !default;

These values will then be compiled to look something like this:

.t-a(l) { text-align: left; }
.t-a(c) { text-align: center; }
.t-a(r) { text-align: right; }

These aliases are used consistently throughout the library, so if you would like to update the value of one (e.g., change "c" to "center"), you only need to updated it in one place, and the library will handle the rest.

As a note, you may find that some aliases have the same value. This is because, in an attempt to keep class names as brief as possible, some values are used multiple times. For example, the values of $a-block and $a-bottom are both b. We assure you, this is only done in places where we are sure the values will never be used together.

Context Aliases

Context aliases refer to the values that are used as the context in the class name syntax. These are used to provide a centralized hub to view all the contexts at a glance but also be able to change them in one place. Let's look at the context alias for text-align:

$c-text-align: t-a !default;

This alias will then be used to identify the context of all the text-align classes:

.t-a(l) { text-align: left; }
.t-a(c) { text-align: center; }
.t-a(r) { text-align: right; }

By updating the value of $c-text-align or any other context alias variable, Elassus will automatically change the context of all the corresponding classes.

Class Name Syntax

You may be thinking to yourself, "I love Elassus, but I hate the class name syntax, because parenthesis and colons in class names are dumb. I want something different". That is encouraged and welcome! That's why we made the build-class-name function!

This function is pretty straightforward, in that it takes some arguments that it can then turn in to a class name. So if you want it to return something entirely different, you can either change the data being passed to it, rewrite the function, or both!

For the sake of example, let's rewrite it to return something that just uses dashes instead of parenthesis, colons, and all that.

@function build-class-name($context, $value: null, $breakpoint: null, $pseudo-classes: null, $pseudo-only: false) {
   $result: #{$context};

   @if($value) {
       $result: #{$result}-#{$value};
   }

   @if($breakpoint) {
       $result: #{$result}-#{$breakpoint};
   }

   @if($pseudo-classes) {
        $base: $result;

        @for $index from 1 through length($pseudo-classes) {
            $pseudo-class: nth($pseudo-classes, $index);
            $alias: if((length($pseudo-class) > 0), nth($pseudo-class, 1), null);
            $selector: if((length($pseudo-class) > 1), nth($pseudo-class, 2), null);
            $new: $base;

            @if ($alias) {
                $new: "#{$new}-#{$alias}";
            }

            @if ($selector) {
                $new: "#{$new}:#{$selector}";
            }

            @if ($pseudo-only == true and $index == 1) {
                $result: "#{$new}";
            } @else {
                $result: "#{$result}, .#{$new}";
            }
        }
    }

    @return $result
}
property(value)breakpoint:pseudo

From this rewrite of the function, you will get classes that go from this:

.bd-c(primary),
.bd-c(primary):f:focus,
.bd-c(primary):h:hover,
.bd-c(primary):a:active {
  border-color: #009688; 
}

To this:

.bd-c-primary,
.bd-c-primary-f:focus,
.bd-c-primary-h:hover,
.bd-c-primary-a:active {
  border-color: #009688;
}

That was easy.
